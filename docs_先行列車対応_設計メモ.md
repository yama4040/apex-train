# 先行列車を考慮した運転曲線最適化 — 設計メモ（2026-07-20）

単一列車の駅停車がほぼ達成されたため、次フェーズとして先行列車（将来的には後続列車も）を考慮した運転最適化に取り組む。本メモは方式検討の結論と、実装前に詰めるべき論点をまとめたもの。

## 1. 目標（CLAUDE.mdより）

### 先行列車のみ
- 先行列車が遅延していても**駅間に停車せず**次駅まで走行（停車しない速度で惰行する方策）
- 接近しすぎた場合はCBTC停止限界（先行の50m手前）までに停止できる
- 信号開通後は駅に向かって加速し、駅停車に向けて減速できる

### 先行・後続の両方
- どちらかが遅れている場合、**列車間隔が一定になるよう速度調整**（早めの惰行など）

### 共通
- 間隔保持のための遅延はやむを得ないが最小限に
- 駅間停車は極力回避
- ノコギリ運転なし・省エネ・乗り心地（単一列車時と同様）

### 設定
- 各列車の駅停車時間30秒／CBTC停止限界50m（先行最後尾〜自列車先頭）／1両編成・列車長20m

## 2. 採用方式: 1回LLM呼び出し＋二重蒸留

### 検討した案
- 案A: 1回のLLM呼び出しでモード判定＋評価を同時出力
- 案B: LLM2回呼び出し（1回目モード判定→2回目モード入力で評価）
- 案C: モード判定をPythonルール化＋LLMは評価のみ

### 結論（採用）
**1回のLLM呼び出しで {mode, reward, reason} を同時出力し、蒸留NNを2つに分ける**:

```
【データ生成】 状態 → LLM（1回） → {mode, reward, reason}
【蒸留】      モードNN:  状態(46次元) → mode（4クラスsoftmax）
              評価NN:    状態 + mode one-hot(4)（=50次元） → reward（ゲート＋回帰の2段構成を維持）
【RL実行時】  状態 → モードNN → argmaxモード → 評価NNへ入力 → 報酬
```

### 採用理由
1. **研究仮説との一貫性**: 「LLMは文脈から状況判断できる」という仮説を、評価だけでなく状況判断（モード決定）の蒸留にまで貫徹できる（階層的蒸留）。案Cはこの仮説の半分を放棄する
2. **実行時整合**: RL実行時のモードはモードNN（毎ステップ推論可能）から供給され、学習時（LLMモードラベル→評価NN教師入力）と同一経路。LLMにモード判定させつつ実行時に再現できる唯一の構成
3. **コスト**: 案Bは評価時間・API失敗面が2倍（実測90秒/行前後）。1回呼び出しなら×1のまま
4. **診断性**: モードNNの精度（混同行列）を独立に測定でき、報酬の誤りを「モード判定」と「評価」に切り分けられる。ゲート分類器の成功実績あり、4クラス分類は回帰より易しい

### 運転モード（フェーズ1は3種で開始／将来4種）
通常運転 / 遅延回復モード / 駅間停車防止モード（＋フェーズ2で間隔調整モードを追加）
（定義・優先順位はLLMプロンプト側に明文化。判定はLLM、模倣はモードNN）

**決定事項（2026-07-22）:**
- **モード数**: 間隔調整モードは後続列車前提でフェーズ1ではデータがほぼ0のデッドクラスになるため、フェーズ1は**3モードで学習**する。ただしモードone-hotの次元は**4で確保**しておき、フェーズ2で間隔調整を追加してもモードNNの再学習のみで済ませ、評価NNの入力次元（50次元）は据え置く。
- **優先順位（同時成立時の採用順）**: 安全(制限速度/CBTC) > 間隔調整 > 駅間停車防止 > 遅延回復 > 通常。フェーズ1の実効順は **安全 > 駅間停車防止 > 遅延回復 > 通常**。

## 3. 詰めるべき論点

### ①【最大の未決】後続列車が環境に存在しない
- 現状 `backward_info="後続列車なし"` 固定。間隔調整モードの半分は後続前提
- **フェーズ分割を推奨**: フェーズ1＝先行のみ（駅間停車回避・CBTC・解放後回復）→ フェーズ2＝後続追加
- 後続の実装は先行列車方式（CSV軌道の直接再生・7/17実装）を流用可能。開始条件（自列車の後ろへいつ発車させるか）の設計が必要

### ② 誤りの連鎖（カスケード）
- モードNNの境界誤判定→評価NNが誤ったモード前提で確信的に採点
- 対策: 評価NN入力はハードone-hot（LLMラベル／argmax）で学習・実行を統一。モードNNの混同行列で弱い境界を特定し、そこへグリッドデータを追加する運用（既存手法のモード版）

### ③ モードのチャタリング
- 0.1秒刻みで境界付近のモードが毎ステップ入れ替わると報酬・Q入力が暴れる
- 対策候補: (a)実行時のみ最低保持時間（3〜5秒）、(b)softmaxのEMA平滑
- ただし両方とも「LLM模倣からの逸脱」を含むため、**最初の評価バッチでLLMモードラベルの軌道沿い安定性を実測してから決定**する

### ④ 既存データ（約26,000行）のモードラベル
- 選択肢: (a)全再評価（純度最高・コスト大）／(b)先行なし行は自明なので機械付与（先行なし×delay小→通常等）し、**先行あり行（約2,500行）＋新規データのみLLM評価**
- 現実解は(b)。ただし機械付与の許容範囲は要方針決定

### ⑤ プロンプト設計

## 5. モードの定義メモ（2026-07-22 改訂）

### 前提：2種類の「遅延」指標
モード判定・評価には性質の異なる2つの遅延指標を用いる（[[hover-exploit-and-reward-calibration]]の残タスクとも関連）。

- **駅出発時遅延量（新規・エピソード中固定）**: 標準運転時間180秒に対し、駅出発時点でどれだけ遅れているか。`environment2.py`の`reset(delay=...)`引数から確定できるため、エピソード開始時に`self.departure_delay`として保持し、走行中は固定値として扱う。→ **遅延回復モードの判定に使用**（エピソード単位で安定＝チャタリングしない）。
- **走行中遅延 `current_delay`（既存・連続値）**: 計画ダイヤを過ぎてからの経過秒（残り時間が正の間は0）。走行中に先行へ詰まって遅れが拡大していく状況の検知・スコアリングに使用。

### モード定義（フェーズ1）
優先順位（高い順）: **安全(制限/CBTC) > 駅間停車防止 > 遅延回復 > 通常**。上位モードの条件が成立していれば下位モードには入らない。

1. **通常運転モード**
   - 条件: 駅出発時遅延 < 30秒 かつ 先行列車に塞がれていない（先行なし、または先行との距離・CBTC現示に十分な余裕がある）。
   - 方策: 省エネルギー最優先。早め惰行を許容し、無駄な加減速・ノッチ切り替えをしない。乗り心地重視。

2. **遅延回復モード**
   - 条件: 駅出発時遅延 ≥ 30秒 かつ 駅間停車防止モードが発動していない（＝先行に塞がれていない）。
   - 方策: 省エネルギー性より遅延回復を優先する。惰行を遅らせ制限速度近くまで力行を維持する運転を高評価とする。
   - 制約: 制限速度の超過は不可。ノッチを規定時間以上キープしないのこぎり運転も不可。
   - **通常モードとのスコア差別化（要）**: 通常なら高評価となる「早め惰行・低め巡航」を、このモードでは減点する。制限近くの巡航維持・積極的な力行を高評価とする。この差を明示しないと評価NNが2モードを区別できない（論点②カスケードの逆リスク）。

3. **駅間停車防止モード**
   - 条件: **駅出発時点**で先行列車の状態（位置・速度・在線/出発状況）を見て、このまま通常走行すると駅間に停車してしまうと物理的に予測できる場合。典型例: 駅出発時点で先行列車がまだ次駅に到着しておらず駅到着ブレーキ中／次駅を出発していないため、通常走行だと追いついてCBTCに引っかかり機外停車に至る。優先度は遅延回復より上。
   - **判定タイミング（決定 2026-07-22）**: **駅出発時点で発動を判定し、原則エピソード中保持**（解放条件を除く。下記参照）。「先行が駅到着ブレーキ扱い中→機外停車する前に早めの惰行を高評価」というLLMの物理予測を、駅出発時の先行スナップショットに基づいて行う。発動は一方向（発動→解放）なのでチャタリングは発生しない。
   - **必要な特徴量（論点D・G の解決方針）**: 下記「駅出発時スナップショットの項目」を参照。発動用の固定スナップショットと解放用の走行中現在値の双方をプロンプト・状態に載せる。
   - 方策: 早め惰行で駅間停車を回避する方策を獲得する。停止が避けられない場合は安全（CBTC停止限界50m以内での停止）を最優先とする。
   - **解放条件（決定 2026-07-22）**: **駅出発時に発動・先行クリアで解放**のハイブリッド。駅出発時点の先行スナップショットで発動し、走行中に先行列車が次駅を出発して十分に前方がクリア（CBTC現示に余裕）になった時点で通常/遅延回復モードへ解放する。純粋な駅出発時固定だと、先行が30秒停車後に発車して先行してしまい追いつかないケースで早め惰行が無駄な遅延になる（過保護）ため。
   - **モードNN上の含意**: 遅延回復モードが駅出発時遅延（エピソード固定値）のみをキーにするのに対し、駅間停車防止モードは①発動用＝駅出発時の先行スナップショット（固定特徴量）と②解放用＝走行中の先行クリア指標（CBTC現示速度／先行距離）の**両方**をキーにする。どちらも状態（46次元）に含める必要がある。解放後は駅出発時遅延に応じて通常/遅延回復へフォールスルーする。

### 駅出発時スナップショットの項目（確定 2026-07-22）
モード判定に用いる入力。**駅出発時に確定する固定スナップショット**（発動用）と**走行中の現在値**（解放用・既存）に分かれる。

**A. 駅出発時固定スナップショット（新規追加・エピソード中固定）**
1. **先行列車の遅延秒数** — 遅れ＝正の符号。先行が遅れているほど詰まる。取得方法は先行CSV（`f_train_delay*`）のメタデータ/ファイル名規約から（後工程で実装）。
2. **先行列車の距離** — 車間の定義は CBTC 基準（先行最後尾−自列車先頭、列車長20m補正）に統一（論点⑦）。
3. **次駅を発車済みか** — 対象は「自列車の次の停車駅」。ブール。先行が塞いでいるかの核となるフラグ。
4. **標準運転間隔** — 実装の `fowerd_train_time_offset`（先行が自列車より何秒先に出発したか）。apex2.pyでは `headway=random.uniform(30,120)`。エピソードごとに確定する有意な値。
5. **（既決）自列車の駅出発時遅延量** — 遅延回復モード判定と共通。実効車間 ≈ 標準運転間隔 ＋ 自列車遅延 − 先行遅延 の関係で、機外停車リスク判定に自列車遅延も効くため、モード判定Stepに併せて渡す。

**B. 走行中の現在値（既存・解放用）**
- 現在の先行距離・速度（`forward_info`）、CBTC現示速度（`signal_speed`）。駅間停車防止モードの解放（先行クリア）判定と、走行中の安全担保（上位優先）に使用。

**物理的関係**: 駅出発時の実効車間 ≈ 標準運転間隔 ＋ 自列車遅延 − 先行遅延。この値が小さいほど機外停車リスクが高い。距離単体（先行が駅到着ブレーキ中だと距離は大きいこともある）より、この時間ベースの関係が効く点をプロンプトで明示する。

### LLM推論のStep構造（確定 2026-07-22）
1回のLLM呼び出し内で2段階に分ける（[[hover-exploit-and-reward-calibration]]の段階評価化を踏襲）。
- **前段（モード判定）**: 優先順位（安全 > 駅間停車防止 > 遅延回復 > 通常）に従い、上記スナップショット＋走行中現在値から1モードを選び `mode` を確定。駅間停車防止は「駅出発時は詰まる予測か（発動）」＋「今は先行がクリアしたか（解放）」の両方を判断。
- **後段（評価）**: 選んだモードの評価基準に従って `reward` と `reason` を算出。
- **出力**: `{mode, reward, reason}`。

### 駅間停車防止モードの加速上限評価（target_speed_no_stop）（確定 2026-07-22）
「機外停車せずどこまで加速してよいか」の数値基準を、既存 `required_speed.py` の `calculate_required_speed` を転用して算出する。現行の required_speed は制約が「定刻までに駅到着」だが、これを「**先行がクリアするまで駅に着かない（先行に追いつかない）**」へ差し替える。

```
先行クリア残時間 = 先行列車が自列車の次駅を発車するまでの残り秒数
                  （先行CSVから算出。駅停車時間は先行CSVに織り込み済み）
実効所要時間     = max(定刻残り時間, 先行クリア残時間 + 安全マージン)
target_speed_no_stop = 「その速度から惰行すると実効所要時間ちょうどに次駅到着する速度」を二分探索
                      （※現在速度に依存しない状況ベースの上限。§7の【重要修正】参照。
                        先行が塞いでいない＝実効所要時間が定刻残り時間と同じ時は従来 required_speed に縮退）
```

- 先行が長く塞ぐほど実効所要時間が伸び、target_speed_no_stop は下がる＝低速で惰行トロトロ進めば機外停車しない、という**加速上限**が出る。
- 先行がクリア（先行クリア残時間 ≤ 定刻残り時間）になると自動的に通常の required_speed と一致 → **解放後の再加速が自然に高評価**（CLAUDE.md「信号開通後は駅に向かって加速」を表現）。

**プロンプトへの入力（決定）**: `required_speed`（定刻ベース・従来通り）と `target_speed_no_stop`（先行考慮）の**両方**を渡し、モードで使い分ける。
- 通常運転モード / 遅延回復モード → `required_speed`（従来通り）
- 駅間停車防止モード → `target_speed_no_stop`

**駅間停車防止モードの採点基準**:
| 自列車の操作 | 評価 |
|---|---|
| `target_speed_no_stop` を明確に超える方向への力行 | 減点（機外停車を招く過剰加速） |
| 上限付近まで（超えない範囲で）の加速・維持、超過なら早め惰行で上限へ収束 | 高評価 |
| 上限を大きく下回る過度な低速・惰行しすぎ（ちんたら） | 減点（無駄な遅延。停止回避マージンは許容） |
| CBTC現示が下がり減速している最中のブレーキ操作 | 高評価（衝突回避・その瞬間の正しい操作） |
| 機外停車して先行待ちしている**状態**（速度0・CBTC=0） | 中立 ~0.5（安全だが回避すべきだった事態。§6参照） |

**要決定の残パラメータ**: 実効所要時間の「安全マージン」（先行クリア直後に着かず少し余裕を持たせる秒数）。プロンプト実装・スモークテスト時に調整。

**実装上の含意**: 先行クリア残時間は走行中の連続値（先行位置の関数）。モードNN／評価NNが target_speed_no_stop を再現するには、先行クリア残時間または target_speed_no_stop を状態に含める必要がある（論点⑥マニフェスト方式で吸収）。

### 将来（フェーズ2）: 間隔調整モード

## 6. プロンプト再編方針（既存CBTCルールとの統合）（2026-07-22）

論点⑤の本丸。現行プロンプト（`evaluate_csv_with_llm.py`）の「先行列車がいる場合はCBTC遵守が絶対最優先、機外停車すら高評価」という思想（`:220-221` `:269` `:277`）と、新目標「早め惰行で機外停車を回避するのが上位」の衝突を解消する方針。

### 基本原則: per-step評価を守れば衝突は解ける
現行の大原則「現在のノッチ操作のみを評価（過去のミスも将来の結果も評価しない）」（`:101`）を守ると、両者は衝突しない。**評価するフェーズが違うだけ**である。

| フェーズ | 現在の正しい操作 | モードによる差 |
|---|---|---|
| 加速・巡航（速度・余裕あり） | 基準速度以下で惰行/巡航 | **モード差はここに集約**。基準速度を required_speed（通常）→ target_speed_no_stop（駅間停車防止）に切替。過剰加速を減点・早め惰行を高評価 |
| 減速・CBTC接近時（先行に迫る） | CBTC沿いブレーキ | 全モード共通で高評価（安全・衝突回避）。その瞬間のブレーキ操作自体は正しい |

→ **「機外停車したブレーキ操作」ではなく「機外停車に至る前の巡航フェーズでの過剰加速」を減点する**。これでper-step原則を壊さず、既存の安全ルール（即0.0・CBTC確認Step）は不変のまま維持できる。

### モードによる基準速度の切替
- 通常/遅延回復モード → `required_speed`（定刻ベース・従来通り）
- 駅間停車防止モード → `target_speed_no_stop`（先行考慮・§5参照）

加速フェーズの「力行継続=高評価」は駅間停車防止モードでは target_speed_no_stop で頭打ち。巡航フェーズの「惰行=高評価／基準割れ=ちんたら」も基準速度をモード別に読み替える。

### 機外停車待機状態の評価（決定）: モード別
- **先行への衝突**: 全モードで reward=0.0（不変）。
- **CBTC現示低下中の減速ブレーキ操作**: 全モードで高評価（動いている状態からの正しい衝突回避操作）。
- **機外停車して先行待ちしている状態（速度0・CBTC=0）**:
  - 通常/遅延回復モード → 現行通り高評価（避けられない急減速への正しい対応）。
  - 駅間停車防止モード → **中立 ~0.5 に下げる**。「安全だが回避すべきだった事態」。動機付けを **早め惰行で流して進む(>0.5) > 機外停車待機(0.5) > 衝突(0.0)** の単調にする。ゼロ中心化後 +0.5/step の積み上げによる hover 搾取（[[hover-exploit-and-reward-calibration]]）回避とも整合。
  - → 現行 `:269`「CBTC=0なら reward=1.0」を駅間停車防止モードでは ~0.5 に変更。

### ちんたら運転ルールのモード対応
現行のちんたら減点は「先行列車がいない（または600m超）」を条件にしており（`:174` `:190-194` `:249`）、駅間停車防止モード（先行が塞ぐ）では大半が発火しない。加えて駅間停車防止モードでは target_speed_no_stop までの低速走行は正当（ちんたらではない）。**target_speed_no_stop を明確に下回る過度な低速のみ**ちんたら扱いとする。

### 先行列車解放時の特例との整合
現行「先行列車解放時の特例」（`:292-297`、先行クリア後の再加速は適切）は、駅間停車防止モードの解放（target_speed_no_stop が上昇し再加速が高評価になる）と自然に整合。特例は維持。

### Step構造へのモード判定の挿入（案）
モード判定Stepを安全系Step（即0.0・制限速度・CBTC）の後、フェーズ評価の前に挿入し、フェーズ評価はモード別基準速度を使う。出力に `mode` を追加。
```
Step1 即0.0ルール確認 / Step2 制限速度 / Step3 CBTC
Step4 モード判定（優先順位: 安全 > 駅間停車防止 > 遅延回復 > 通常）
Step5 停止位置達成可能性 / Step6 フェーズ評価（モード別基準速度）
Step7 ノコギリ / Step8 運転間隔 / Step9 定時性・省エネ（モード別重み）
Step10 総合評価
```
出力: `{mode, reward, reason, checks}`（`mode` はchecksと別のトップレベル項目として追加。checksの8キーは不変）。
