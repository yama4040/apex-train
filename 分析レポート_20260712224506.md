## 1. DQNの現状

### 1.1 改善が確認できた点

**駅手前ホバリング搾取は消滅した。** 前回run（data/20260712001009）で観測された「駅65m手前で速度0.3km/hを維持し0.1秒毎に力行⇔ブレーキを反転して報酬を稼ぐ」行動は、今回のテスト走行には見られない。

- 報酬スケールの動作確認: 駅手前100m内の平均報酬 **0.010**（レンジ0.00〜0.09）に対し巡航区間 **0.716**。実時間あたり報酬密度10倍の搾取源は解消。
- クリープ走行（速度0.5km/h以下での滞留）は終了条件⑤で即打ち切りされる。

### 1.2 新たに顕在化した問題: オーバーラン方策への収束

テストケース終端の状態（dist=最終的な駅までの残距離、v=最終速度[km/h]）:

| cycle | 傾向 |
|---|---|
| 50 | 16ケース中9ケースが **速度32〜57km/hのまま駅を通過**（オーバーラン終了） |
| 100 | 通過と低速クロールが混在（tc11は切替451回・タイムオーバー） |
| 150 | 大半が **速度44〜64km/hで駅を通過**。停止試行がほぼ消失 |

代表例（cycle150 tc0）: 駅の約100m手前からブレーキを開始するが、48km/hからでは間に合わず45km/hで駅を通過して終了。

**メカニズム（旧NNの較正崩れが直接原因）:** cycle150 tc0の減速フェーズ（駅手前400m以内）で、旧NNが与える報酬は

- ブレーキ中: 平均 **0.045**
- 惰行中: 平均 **0.377**

つまり「正しくブレーキして停車に向かう」より「惰行で高速のまま突っ込む」方が約8倍の報酬レートになっている。本来のLLM評価ルールでは適切なタイミングのブレーキは0.8〜1.0だが、旧NNは減速フェーズのブレーキを大幅に過小評価する。ホバリングという搾取経路を塞いだ結果、次に報酬が高い経路である「巡航報酬を最後まで稼いで駅を通過」に方策が流れた。

さらに、停車完了時の高報酬（プロンプト上は誤差1m以内で1.0）は、旧データセットに停車完了フェーズが31行しかないためNNがほぼ学習しておらず、加えて駅直前は0.1秒ステップで報酬が0.1倍にスケールされるため、「停車する動機」が実質的に存在しない。

### 1.3 残存する問題: 先行列車ケースの低速クロール

cycle100 tc11では、駅33m手前まで約2〜7km/hのクロールで進み、切替451回・240秒のタイムオーバーで終了。終了条件⑤（速度0.5km/h以下）と⑥（先行列車ありの場合は10秒5m未満のみ）の適用範囲外をすり抜けている。先行列車がいるケースでは信号待ちと区別が必要なため判定を緩くしており、そこが抜け道になっている。

### 1.4 方策の不安定性

cycle50→100→150で終端挙動が大きく振動しており（高速通過→クロール混在→高速通過）、報酬地形が「良い運転」を指していない状態での学習は収束しない。**旧NNのままこのrunを続けても精密停車は構造的に学習できない**と判断する。

## 2. 評価用NN（現用＝旧回帰器）の質

| 指標 | 旧モデル（現用） | 新方式（実装済み・未配備） |
|---|---|---|
| ラベル0.0への報酬リーク（予測>0.25） | 15.7% | **1.4%** |
| ラベル0.0の平均予測 | 0.082 | 0.012 |
| ラベル1.0の平均予測 | 0.889 | 0.941 |
| 全体MAE | 0.110 | 0.043 |

- 旧モデルの欠陥は逆頻度サンプル重みの極端さ（0.7ビンの重みが0.0ビンの約156倍）で、0.0の大集団での誤差がほぼ罰されないこと。
- 新方式＝「0.0か否か」のゲート分類器＋非0.0のみの回帰器（ハードルモデル）＋sqrt緩和重み。`train_reward_network2.py` に実装済みで、実行すれば `direct_reward_gate2.h5` を含む新モデル一式が生成される。
- ただし上記の数値は**旧プロンプトのデータセットでの検証値**。減速フェーズのブレーキ過小評価（1.2節）はデータセット自体の偏りに起因するため、新方式の配備だけでなく改訂プロンプトでの再評価が必要。

## 3. 学習データセットの質（train_reward_csv_direct、8ファイル・8,221行）

| ラベル | 件数 | 割合 |
|---|---|---|
| 0.0 | 4,149 | 50.5% |
| 0.1〜0.9（中間） | 1,318 | 16.0% |
| 1.0 | 2,754 | 33.5% |

- **二極化が深刻**: 0.0と1.0で全体の84.0%。旧プロンプトの「即0.0ルール」拡大が原因（改訂済みプロンプトで再評価すれば中間ラベルが復活する見込み）。
- **停車完了フェーズが31行しかない**（0.0が10行・1.0が21行）。停止の価値をNNが学習するにはデータが桁違いに不足。
- 減速フェーズ×ブレーキは1,534行あり、うち高評価（≥0.8）は485行。ブレーキ成功例が皆無なわけではないが、旧プロンプトでは「早すぎるブレーキ」が一律0.0のため、境界付近の勾配情報が失われている。
- **分布外（OOD）問題**: 今回のrunで新たに出現した「高速のまま駅に突入する」挙動の近傍データは評価済みログにほとんど含まれず、NNの予測が信頼できない領域で方策が学習されている。

## 4. 今後の方針（優先順）

### ① 現行runの停止（推奨・即時）
旧NNの報酬地形では「惰行0.377 vs ブレーキ0.045」のため、続けてもオーバーラン方策が強化されるだけ。計算資源の無駄になる。

### ② 改訂プロンプトでのデータセット再評価
- `評価用csv` を改訂済み `evaluate_csv_with_llm.py` で再評価する。
- **今回のrunの新挙動を評価対象に加えること**: 高速駅通過（cycle150のtc0等）や先行列車ケースのクロール（cycle100のtc11等）のLLM評価用CSVを評価対象に追加し、「オーバーラン確実の惰行=0.0」「正しいブレーキ=0.8〜1.0」の教師信号でOOD領域を埋める。
- 停車完了フェーズの不足を補うため、正しく停車できた軌道（過去runの成功例や手動・ルールベース運転の軌道）を意図的に収集して評価に混ぜるのが望ましい。

### ③ 新方式NNの学習・配備
```
MPLBACKEND=Agg python train_reward_network2.py
```
で回帰器・ゲート分類器・スケーラーの3点が生成され、`direct_reward_predictor2.py` が自動でゲート付き推論に切り替わる。配備後、「減速フェーズでブレーキ報酬＞惰行報酬になっているか」をスポットチェックしてからDQNに進むこと（前回作成の検証手順: ラベル0.0のリーク率、フェーズ×ノッチ別の平均予測）。

### ④ DQN再学習と監視指標
再学習時は以下を50サイクル毎のテストで監視する:
- 停止位置誤差（最終dist）が縮小傾向にあるか
- 減速フェーズの切替回数（ノコギリの再発チェック）
- 終端の速度（オーバーラン方策の再発チェック）
- history_0.csv の報酬合計（スケール変更後の値は過去runと比較不可な点に注意）

### ⑤ ④で停車が学習されない場合の追加策
報酬がすべて非負である限り「エピソードを長引かせる」誘因は残る。その場合は次を検討:
- **報酬のゼロ中心化**（`reward = llm_reward - 0.5` 等。NN出力の線形変換であり「報酬はLLM蒸留NN出力のみ」の方針と両立し得る）
- 目標達成時の終端ボーナス（方針の緩和が必要）
- 駅手前の優先度補正（apex2.pyのpriority_correction）が「駅手前の遷移」を過剰に学習させていないかの見直し

### ⑥ 先行列車ケースのクロール対策
終了条件⑥の先行列車ありの分岐（10秒5m未満のみ）が抜け道になっている。「先行列車がいても、CBTC信号現示が十分高い（例: 20km/h以上）のに平均速度が閾値未満」なら停滞とみなす、等の条件追加を検討する。

### ⑦ 中期的なデータセット拡充方針
- 目標は「二極化率84%→60%台」「停車完了フェーズ数百行」「減速フェーズのブレーキ高評価例の拡充」。
- 再評価のたびに `check_reward_distribution.py` 等でラベル分布を確認し、偏りが再発したらプロンプトの閾値（段階減点の幅）を調整する。
