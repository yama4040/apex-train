# 引継ぎ資料 — Apex-Train × LLM蒸留報酬 研究（2026-07-20時点）

## 1. 研究の枠組み

「LLMは複雑な環境でも文脈から状況を判断し適切な評価を下せる」という仮説のもと、
1. シミュレータ走行ログをLLMにノッチ操作評価させてデータセットを作成（`evaluate_csv_with_llm.py`）
2. LLM評価を蒸留したNNを報酬関数として構築（`train_reward_network2.py` → `direct_reward_model2.h5`等）
3. Apex DQN（`apex2.py`＋`environment2.py`）で列車運転方策を学習
という3段パイプラインで研究している。**報酬はLLM蒸留NNの出力のみ**（環境側の報酬加算は禁止）が設計制約。

## 2. 現在の主要な設計（確定事項と経緯）

### 報酬設計（environment2.py）
- **ゼロ中心化**: `reward = (llm_reward − 0.5) × (time_step / base_time_step)`。順序重要（中心化→時間スケール）
  - time_stepスケール: 駅手前100mで0.1秒ステップになるため、スケールしないと報酬密度10倍→駅前滞留の搾取（2026-07-12実証）
  - ゼロ中心化: 全報酬非負だと「エピソードを長引かせる」のが常に得＝時間稼ぎ系搾取の温床。導入によりQ過大評価（駅前で最大8,600倍）も解消し、Qmax≒実現リターンに（2026-07-14実証）
- **終端報酬のイベント化**: goal_reached時のみ `(llm−0.5)×1.0` 満額（時間スケールを外す）。停止精度の差が0.08→0.8になり±1mへの勾配が生まれる

### 終了条件（environment2.py、番号はコードのコメント準拠）
- ① タイムオーバー: 標準運転時間+60秒
- ② 目標達成: 駅−10m〜+5mで**速度0.5km/h以下**（0.0だと準停車が⑥に殺される）
- ③ 先行列車異常接近: **40m**未満で停止（50mはCBTC停止限界＝正着位置なので失敗にしてはいけない）
- ④ オーバーラン: +5m超で走行中
- ⑤ 実質停止: 駅10m超手前でv≤0.5（クリープ搾取封じ）。先行列車100m以内は信号待ちとして除外
- ⑥ 停滞検出: 駅400m以内は10秒5m未満／それ以外30秒25m未満。**駅20m以内は適用外**（±1mに必要な低速最終進入を殺さないため）

### フェーズ判定の同期（重要）
- 停車完了フェーズ: 駅10m以内 かつ **v≤0.5**（goal条件と同期。environment2._get_current_phase_str と apex2.py テスターの両方）
- ズレると終端報酬が「減速フェーズ×ブレーキ」（|Δ|≤5一律高評価）で採点され停止精度が効かなくなる（2026-07-18実証）

### 報酬NN（2段階×46次元）
- **ハードルモデル**: ゲート分類器（0.0か否か）＋非0.0のみの回帰器。推論はゲートP(0.0)≥0.5→0.0、それ以外は回帰値を[0.1,1.0]クリップ（`direct_reward_predictor2.py`）
- **クリップ済み特徴（46次元の核心）**: `margin_stop_dist_clip=clip(Δ,±30m)`・`dist_to_station_clip=clip(dist,±20m)`・`dist_to_station_clip3=clip(dist,±3m)`。生のmargin/distはσ≈775〜793mで、±5mや±1mの停止境界が0.0064〜0.1σになりNNから**原理的に見えない**。クリップで可視化（σ15m/10m/0.9m）。この発見が精密停車の突破口だった
- **サンプル重み**: ビン重み（1/√件数）×停車完了フェーズ`STOP_PHASE_WEIGHT=3.0`（回帰器・ゲート両方、平均1.0再正規化）
- **世代互換**: predictorとanalyzeはスケーラーの`n_features_in_`で43/45/46次元を自動判別（旧モデルでも動く）
- 3ファイル同期の掟: 特徴量の追加は `train_reward_network2.py`・`direct_reward_predictor2.py`・`analyze_reward_nn_vs_llm.py` を**必ず同時に**同じ並び順で変更

### 先行列車（environment2.py）
- **CSV記録軌道（position/speed）の線形補間つき直接再生**（set_states方式）。アクション開ループ再生は初期丸め＋バンバン制御感度で発散するため廃止（197m乖離を実証）
- 初期位置は `int(headway + delay)` 行目。CSV超過後は最終行で保持
- 過去の凍結バグ（非整数開始時刻で一切動かない）は修正済み

### 制限速度の駅通過後フォールバック
- 駅を数cm過ぎるとfront_sectionsが空→制限0→CBTC現示0→「信号超過0.0」になるバグがあった。`current_speed_limit`が最終非ゼロ値をキャッシュして維持

## 3. 搾取（reward hacking）の歴史と教訓

方策は毎回、報酬地形の「教師データが無い最寄りのポケット」に逃げ込む。時系列：

| 時期 | 搾取 | 原因と対策 |
|---|---|---|
| 7/12 | 駅手前ホバリング（0.1秒毎に力行⇔ブレーキ） | 報酬密度10倍＋NNリーク → time_stepスケール・2段階NN |
| 7/12 | 高速オーバーラン | 旧NNがブレーキ0.045 vs 惰行0.377と逆評価 |
| 7/12-13 | 巡航ブレーキ低速化ループ | 「巡航×ブレーキ×低速」がデータ0行の空白→ゲートが確信的誤判定 |
| 7/13-14 | 駅前クロール | NNが停止を評価できない＋Q過大評価 → ゼロ中心化 |
| 7/15-17 | 惰行のまま駅通過（31→38→47km/hと毎回別の速度帯へ） | 軌道データは(v,dist)平面の「線」しか埋めない → **グリッド合成データで「面」を埋める**方式へ |
| 7/19 | ダイブ農法（巡航で自らブレーキ→再力行1.0をfarm） | 「必要速度未満なら力行高評価」が自作自演を見抜けない → prev_notch条件のプロンプトルール追加 |

### 教訓（運用原則）
1. **DAggerループ**: runごとに方策が見つけた新挙動をLLM評価してデータセットへ追記→NN再学習。静的な蒸留報酬モデルは必ず盲点を突かれる
2. **グリッド合成データ**: 「エージェントが一度もやらない正解行動」（適正ブレーキ・停車完了等）はログからは採れない。状態を格子生成しLLMにラベル付けさせる
3. **プロンプト変更は必ず実APIスモークテスト**（数行）してから本評価へ。新ルールは隣接フェーズへの巻き込み事故を起こす（7/20は正しくなるまでガード3回）
4. **境界には両側からコントラストデータ**を densely 置く（1m段差=0.05m刻み等）
5. LLM評価のゆらぎ・旧基準ラベル・APIエラー混入は定期的に**精査→退避**（§5）

## 4. 定型運用手順

```bash
# 1) 評価（プロンプト改訂があれば全PCで最新版を確認してから）
python evaluate_csv_with_llm.py     # 評価用csv/ → 評価済ログ/llm_evaluated_dataset.csv（毎回上書き新規！）
# 2) 出力をデータセットへ
cp 評価済ログ/llm_evaluated_dataset.csv train_reward_csv_direct/llm_eval_<日付>.csv
# 3) 評価済みファイルは 評価用csv/評価済みアーカイブ/ へ移動（二重評価防止）
# 4) 再学習（±3分。plt.show対策でMPLBACKEND必須）
MPLBACKEND=Agg python train_reward_network2.py
# 5) プローブ検収（§6）に合格したら
# 6) 新runを開始（環境・NNの変更はrun起動時にしか反映されない）
python apex2.py
```

### 落とし穴
- `evaluate_csv_with_llm.py` は起動時に出力を**上書き新規作成**。実行前に `pgrep -f evaluate_csv_with_llm` で既存プロセス確認（過去に並走で消し合う事故）
- 複数PC分散評価時はプロンプトの版ズレに注意（旧版ラベルが混入した実績あり）
- 稼働中のrunにNNモデルを上書きすると、1000サイクル毎のActor再生成時に報酬関数がすり替わる。再学習は新runとセットで
- 評価失敗行はreward=Noneで退避される設計だが、旧版スクリプト経由でエラーHTMLがreward=0.0のまま混入した実績あり（退避基準でスキャン可能）

## 5. データセット管理

- 本体: `train_reward_csv_direct/*.csv`（`required_speed`列がないファイルは学習時スキップ）
- 退避: `train_reward_csv_direct/退避_旧基準ラベル/`（元ファイル無傷＋`conflict_rows__*.csv`＝退避行。LLM評価フォーマットのままなので後で再評価回収可能）
- 退避基準（analyze同梱ではなくアドホックスクリプトで実施してきた。カテゴリ）: 旧ブレーキ段階（Δ帯×低ラベル）／駅100m以内の調整ブレーキ0.0／停車完了の段階評価矛盾／APIエラー混入／制限速度0バグ由来
- 合成データ生成の定型: 21列のLLM評価フォーマット（req_stop_dist・required_speed・delta_stopは評価スクリプトが再計算するのでプレースホルダ可）。生成後は必ず全行のパース＋プロンプト生成検証

## 6. 検収プローブ（再学習のたびに実施）

`DirectRewardPredictor`に状態辞書を入れて確認する定番セット：
1. **ブレーキ判断マップ**: v×Δで階段（Δ−10→0.0／0→0.8-1.0／+10→0.5-0.7／+30→0.3-0.5）
2. **停車完了マップ**: 誤差0.4m→1.0／1.5m→0.8／4m→0.5／7m→0.2／12m→0.0（v=0とv=0.3〜0.5の両方で）
3. **惰行接近**: Δ>0→0.8+／Δ<−5→0.0〜0.1（距離依存が消えていないか）
4. **搾取チェック**: 巡航×力行×直前ブレーキ長→0.1〜0.3／正当力行（直前惰行）→0.9-1.0／CBTC解放回復→高／駅出発力行→1.0
5. 全体MAE・ラベル0.0リーク率（analyze_reward_nn_vs_llm.pyで全体検証も可）

## 7. 診断ツール

- `analyze_reward_nn_vs_llm.py`: NN vs LLMラベルの分布・ヒートマップ・ルールベース検証（現行プロンプトの閾値と同期済み。2段階NN・46次元・世代判別対応）
- `analyze_qnet_coverage.py`: Qテーブルの埋まり具合（maxQ／貪欲方策マップ／行動間ギャップの3面ヒートマップ、`--overlay-csv`で実軌道重ね描き）
- Q較正の手動チェック: テスターCSV（38列）でQmaxと実現割引リターンを距離帯別に比較（乖離が大きければ学習不安定）
- テスターCSV: 8 raw + 25 norm + Q3 + step_reward + llm_reward の38列。`LLM評価用/`サブフォルダに21列のLLM形式も同時出力
